			  Since you have 4 directions, you have a 4-bit number. The maximum value is <!---->1111 in binary, which is **15** in decimal or **F** in hexadecimal.

			- ### 1. The Mapping (Bitmasking)
			  
			  To understand this, you need to look at **Bitmasking**. This is the standard way programmers store multiple True/False values in a single number.
			- **North (N):** 20=1 (Binary: `0001`)
			- **East (E):** 21=2 (Binary: `0010`)
			- **South (S):** 22=4 (Binary: `0100`)
			- **West (W):** 23=8 (Binary: `1000`)
			  
			  **How to calculate a cell value:**
			  If a cell has a wall on the **North** and **South**, you add their values: 1+4=5.
			- In binary: `0101`
			- In hex: `5`
			  
			  If a cell is completely enclosed (the "42" pattern cells): 1+2+4+8=15.
			- In binary: `1111`
			- In hex: `F`

			- ### 3. Practical Implementation Tips
			- #### Setting a wall (Bitwise OR)
			  
			  When your generator decides a wall should exist, you "OR" the bit into the cell's current value:
			  
			  ```
			  # Start with an empty cell (0)
			  cell_value = 0 
			  
			  # Add a North wall (1)
			  cell_value |= 1 
			  
			  # Add a West wall (8)
			  cell_value |= 8
			  
			  # cell_value is now 9 (Binary 1001)
			  ```
			- The "Coherent" Data Rule
			  
			  Your project instructions say: *"each neighbouring cell must have the same wall if any."*
			  This is the trickiest part of bitwise logic in mazes. If you set an **East** wall for cell (0,0), you **must** also set a **West** wall for cell (1,0).
			- #### Printing to Hex

			  Python

			  print(format(value, 'X')) # Outputs: 'A'

	1. The Theory: Bitwise Maze Generation

Most people think of a maze as "walls" and "paths." But in your code, every cell is a single integer. We use Bitmasks to represent the walls.

Imagine a 4-bit number where each bit represents a compass direction:
    Bit 0 (Value 1): North
    Bit 1 (Value 2): East
    Bit 2 (Value 4): South
    Bit 3 (Value 8): West

If a cell has the value 15 (1+2+4+8), it means all four walls are closed (1111 in binary). If we "carve" a path North, we subtract 1. Now the value is 14 (1110), meaning the North wall is gone!

1. The Power of Two (Binary Alignment)

If we used regular numbers like 1,2,3,4, we would run into a problem. If a cell had a value of 3, does that mean it has a wall at direction #3, or does it mean it has walls at direction #1 and direction #2? It's ambiguous.

By using powers of two, each direction gets its own unique column in binary:
Direction	Decimal	Binary	Position
North	1	0001	1st bit
East	2	0010	2nd bit
South	4	0100	3rd bit
West	8	1000	4th bit

Because of this, any combination of walls creates a unique sum.
    North + East = 1+2=3 (0011). There is no other way to get the number 3 using these powers.
    All walls (15) = 1+2+4+8=15 (1111).

2. How is the "Order" Decided?

The specific assignment (North=1, East=2, etc.) is actually arbitrary—meaning the programmer (you!) decides it at the start. However, most programmers follow the Clockwise Rule:
    North (Top)
    East (Right)
    South (Bottom)
    West (Left)

This makes the logic easier to visualize. If you decided North was 8 and West was 1, the maze would still generate perfectly fine, as long as you were consistent everywhere in your code.

3. Bitwise Math vs. Regular Math
You asked why we use these bits. It allows us to use Bitwise Operators, which are much faster and cleaner for "checking" walls than using a bunch of if/else statements.
    The Check (&): To see if a wall exists, we use "AND".
    if val & 1:
    This looks at the binary and asks: "Is the 1st bit flipped on?" It ignores all other bits.
    The Carve (-= or ^): To remove a wall, you just subtract its value.
    If you are at 15 (all walls) and move North, 15 - 1 = 14.
    Binary: 1111 → 1110. The North bit is now "Empty."

4. The "15" Magic Number

This is why your grid starts at 15. It’s the binary way of saying: "Every single direction is blocked."
As your generate function runs, it picks a wall and "knocks it down" by subtracting that specific bit value.
Teacher's Pro-Tip: If you wanted to add a "Upstairs" and "Downstairs" to your maze (3D), you would just keep doubling the numbers! Up = 16, Down = 32. Binary math handles it perfectly.
